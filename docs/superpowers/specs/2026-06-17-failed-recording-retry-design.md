# 浮声失败录音重试队列设计

日期：2026-06-17

## 背景

浮声当前会在部分失败路径保存文本草稿：LLM 整理失败时保存原始 ASR 文本，粘贴失败或无输入框时保存整理后的文本。但如果 ASR 接口失败、超时或服务端报错，应用通常拿不到可用文本，用户刚录制的语音会丢失。

本设计新增一个独立的“失败录音”队列，用于保存接口阶段失败的录音和恢复上下文。用户可以在后台页面手动重新请求，重试成功后复制结果到剪贴板并写入普通草稿历史。

## 已确认决策

- 新增独立窗口“失败录音”，不合并到现有“草稿历史”。
- 重试从失败位置恢复：
  - ASR 失败：使用保存的音频重新 ASR，再进行 LLM 整理。
  - LLM 失败：使用保存的 ASR 文本直接重新整理，不重新 ASR。
- 重试成功后：
  - 将整理文本复制到剪贴板。
  - 保存一条普通草稿历史。
  - 从失败录音列表移除。
  - 删除对应音频文件。
- 失败录音队列不受“保留历史草稿”开关影响。
- 队列最多保留最近 50 条，超出时自动删除最旧记录和音频文件。
- 只保存接口阶段失败：ASR 网络/超时/服务端错误，以及 LLM 整理失败。
- ASR 返回空文本、粘贴失败、剪贴板失败、权限失败不进入失败录音队列。

## 范围

### 包含

- 录音期间将音频流同步写入本地 PCM 文件。
- 新增失败录音 SwiftData 模型和存储服务。
- 新增本地音频文件存储服务。
- ASR 或 LLM 接口失败时保存可重试任务。
- 新增“失败录音”窗口、菜单入口、刷新和前置窗口行为。
- 支持手动重试和删除。
- 重试成功后复制到剪贴板、保存普通草稿、删除失败任务和音频。
- 自动裁剪最近 50 条失败录音。
- 单元测试覆盖失败保存、重试、删除、裁剪和窗口入口。

### 不包含

- 自动后台重试。
- 云端同步失败录音。
- 播放失败录音。
- 重新粘回失败时的原输入框。
- 用户自定义失败录音保留数量。
- 保存权限/API Key 缺失导致的失败任务。

## 数据模型

新增 `FailedRecordingRecord` SwiftData 模型。它只表示可重试的失败录音任务，不表示普通文本草稿。

字段：

- `id: UUID`
- `idSortKey: String`
- `createdAt: Date`
- `sourceAppName: String`
- `modeRawValue: String`
- `asrModel: String`
- `polishModel: String`
- `failureStageRawValue: String`
- `errorSummary: String`
- `audioFilePath: String`
- `rawASRText: String`
- `retryStateRawValue: String`
- `lastRetryAt: Date?`

新增枚举：

- `FailedRecordingStage`
  - `asr`
  - `polish`
- `FailedRecordingRetryState`
  - `idle`
  - `retrying`
  - `failed`

`rawASRText` 的含义：

- ASR 失败时为空。
- LLM 失败时保存已经成功识别出的原始 ASR 文本。

## 存储服务

新增 `FailedRecordingStoring` 协议和 `FailedRecordingStore` 实现。

职责：

- 新增失败任务。
- 查询最近失败任务。
- 更新重试状态、错误和最近重试时间。
- 删除失败任务。
- 新增任务后裁剪到最近 50 条。
- 删除记录时通知失败录音窗口刷新。

新增 `FailedRecordingAudioStoring` 协议和 `FailedRecordingAudioStore` 实现。

职责：

- 为当前录音创建临时 PCM 文件。
- 追加写入 PCM 音频块。
- 录音正常成功时删除临时文件。
- 接口失败时将临时文件转为失败录音文件。
- 删除失败任务时删除对应音频文件。
- 重试时读取音频文件并生成 ASR 可消费的音频流。

音频文件目录：

```text
~/Library/Application Support/Fusheng/FailedRecordings/<id>.pcm
```

## 录音保存流程

录音开始后，Coordinator 获取 `AudioRecorder` 的 PCM 音频流，并包装成一个分发流：

1. 音频块继续发送给 ASR。
2. 同一份音频块追加写入本地临时 PCM 文件。
3. 音频电平通知继续由 `AudioRecorder` 负责，不进入失败录音模块。

正常成功时：

- ASR 成功、LLM 成功、输出阶段完成后，删除临时音频文件。
- 不创建失败录音任务。

ASR 接口失败时：

- 保留音频文件。
- 创建 `failureStage = asr` 的失败任务。
- `rawASRText = ""`。
- `errorSummary` 保存 ASR 错误摘要。
- 主流程进入失败状态，显示原 ASR 错误。

LLM 接口失败时：

- 保留音频文件。
- 创建 `failureStage = polish` 的失败任务。
- `rawASRText` 保存已经识别出的文本。
- `errorSummary` 保存 LLM 错误摘要。
- 保留当前已有行为：如果焦点在输入框，可以写回原始 ASR 文本；普通草稿历史仍可保存原始 ASR 文本。

不进入失败录音队列的场景：

- API Key 缺失。
- 麦克风权限不足。
- 麦克风设备或录音器启动失败。
- ASR 成功返回空文本。
- LLM 成功后粘贴失败。
- 剪贴板复制失败。
- 辅助功能权限不足。

这些场景要么没有可重试音频，要么已有文本草稿兜底，或者不是接口请求失败。

## 重试流程

失败录音页面每条记录提供：

- “重新请求”
- “删除”

点击“重新请求”：

1. 检查 API Key。
2. 检查音频文件是否存在。
3. 将该条状态设为 `retrying`。
4. 根据失败阶段恢复：
   - `asr`：读取 PCM 文件，重新 ASR，然后 LLM 整理。
   - `polish`：跳过 ASR，直接用 `rawASRText` 重新 LLM 整理。
5. 整理成功后复制结果到剪贴板。
6. 保存一条普通草稿历史。
7. 删除失败录音记录和音频文件。
8. 刷新失败录音窗口和草稿历史窗口。

重试失败：

- 不删除失败任务。
- 更新 `retryState = failed`。
- 更新 `lastRetryAt`。
- 更新 `errorSummary` 为本次失败原因。
- “重新请求”按钮恢复可用。

复制剪贴板失败：

- 如果已经生成整理文本，则保存普通草稿历史。
- 失败录音记录保留。
- 更新错误为“文本已生成但复制失败”。
- 用户可以从草稿历史复制结果，或处理剪贴板问题后再重试。

重试成功后不尝试自动粘回当时的输入框，因为原输入焦点不可可靠恢复。

## 页面与入口

新增窗口：

- 标题：“失败录音”
- 菜单栏入口：“打开失败录音”
- 打开窗口时主动刷新并前置窗口。

列表排序：

- 按创建时间倒序。
- 同一时间使用 `idSortKey` 倒序保证稳定排序。

每条显示：

- 失败阶段：识别失败 / 整理失败。
- 错误摘要。
- 来源 App。
- 创建时间。
- 整理模式。
- ASR 模型和整理模型。
- LLM 失败时显示 ASR 文本预览。
- 最近重试时间。
- 重试状态。
- “重新请求”和“删除”按钮。

音频文件缺失时：

- 显示“音频文件缺失”。
- 禁用“重新请求”。
- 保留“删除”。

自动裁剪最近 50 条：

- 新增失败任务后执行。
- 删除最旧记录时同步删除对应音频文件。
- 不需要弹窗提示。

## 状态与通知

新增通知：

- `failedRecordingQueueDidChange`

触发场景：

- 新增失败录音。
- 删除失败录音。
- 重试状态变化。
- 重试成功移除失败录音。
- 自动裁剪。

失败录音窗口订阅该通知并刷新列表。

## 与现有模块关系

`AudioRecorder` 继续只负责采集和转换音频，不直接知道失败队列。

`AppCoordinator` 负责：

- 在录音流程中包装音频流。
- 在接口失败时创建失败任务。
- 正常成功时清理临时音频。

`DraftStore` 继续只保存普通文本草稿，不保存失败录音音频。

新增 `FailedRecordingRetryService` 负责从失败任务恢复请求，避免把重试逻辑塞进 SwiftUI View。

`FailedRecordingView` 只负责展示和触发动作，不直接调用 ASR/LLM/剪贴板细节。

## 错误处理

- 保存失败录音本身失败时，主流程保留原接口错误，不用保存失败覆盖原错误。
- 保存失败录音失败要写入日志，便于排查。
- 重试时 API Key 缺失，记录保留，错误更新为“缺少 API Key”。
- 重试时音频文件缺失，不发起请求，按钮禁用。
- 重试失败不删除音频。
- 删除失败任务时，如果音频文件删除失败，仍删除 SwiftData 记录，并记录日志。
- 普通成功路径必须删除临时音频文件，降低隐私和磁盘风险。

## 测试计划

### Store 测试

- 保存失败录音后可按倒序读取。
- `asr` 和 `polish` 失败阶段可往返序列化。
- 重试状态和最近重试时间可更新。
- 删除记录时触发刷新通知。
- 超过 50 条时删除最旧记录。
- 自动裁剪时删除对应音频文件。

### 音频文件测试

- 可以创建临时 PCM 文件并追加音频块。
- 可以将临时文件提交为失败录音文件。
- 可以读取失败录音文件生成音频流。
- 删除缺失文件不导致崩溃。

### Coordinator 测试

- ASR 接口失败时保存失败录音，包含音频路径、模型、模式和错误信息。
- LLM 接口失败时保存失败录音，并保存 `rawASRText`。
- ASR 返回空文本不保存失败录音。
- 粘贴失败不保存失败录音，只保存普通草稿。
- 正常成功后删除临时音频。
- 保存失败录音失败时仍显示原接口错误。

### Retry 测试

- ASR 阶段重试成功会重新 ASR、重新整理、复制剪贴板、保存草稿、删除失败任务和音频。
- LLM 阶段重试成功会跳过 ASR，只重新整理、复制剪贴板、保存草稿、删除失败任务和音频。
- 重试失败会保留任务并更新错误。
- API Key 缺失时不发请求，任务保留。
- 音频文件缺失时不发请求，任务可删除。
- 复制剪贴板失败时保存草稿，失败任务保留并更新错误。

### UI/入口测试

- 菜单栏包含“打开失败录音”入口。
- 打开失败录音窗口会刷新列表并前置窗口。
- 重试中按钮禁用。
- 音频文件缺失时“重新请求”按钮禁用。

## 验收标准

- ASR 或 LLM 接口失败后，用户能在“失败录音”窗口看到对应失败项。
- ASR 失败项能使用保存音频重新识别和整理。
- LLM 失败项能跳过 ASR 直接重新整理。
- 重试成功后文本进入剪贴板和普通草稿历史。
- 重试成功后失败项和音频文件被删除。
- 失败项最多保留最近 50 条。
- 普通录音成功路径不会残留音频文件。
- “保留历史草稿”关闭时，接口失败仍保存失败录音。
