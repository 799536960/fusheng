# 浮声 macOS 语音输入软件方案

日期：2026-06-16

## 1. 产品定位

**浮声** 是一个 macOS 菜单栏语音输入工具。它的目标不是继续依赖微信输入法的语音输入，而是提供一个独立的系统级语音输入能力：

- 有输入框时，语音内容自动输入到当前输入框。
- 没有输入框时，语音内容先在后台记录为草稿。
- 识别结果可以自动整理成更自然、干净、可直接使用的文本。

核心体验是：用户按下快捷键，说话，软件自动完成录音、识别、整理和输入。

## 2. 背景问题

微信输入法的语音输入功能很好用，但它必须依赖当前已有激活的输入框。这个机制带来的问题是：

- 用户必须先点击某个输入框，才能开始语音输入。
- 如果用户正在看设计图、看代码、浏览页面或对照资料，输入焦点会限制操作。
- 一旦用户切换到没有输入框的场景，语音输入就没有地方接收文本。

因此，浮声要解决的问题是：**语音输入不应该被某一个输入框绑定。**

## 3. 核心交互方案

用户打开浮声后，可以在设置中配置一个全局快捷键。

当用户按下快捷键时：

1. 浮声开始录音。
2. 录音音频发送到阿里百炼 ASR 服务进行语音识别。
3. ASR 返回的原始文本再交给大模型整理。
4. 整理后的文字根据当前系统焦点决定去向。

### 3.1 当前有输入框

如果当前前台 App 存在已聚焦的文本输入控件，例如：

- 浏览器输入框
- 微信聊天输入框
- 代码编辑器文本区域
- 备忘录、文档、Notion、飞书等编辑区域

则浮声将整理后的文字直接输入到当前输入框。

推荐实现方式是：

- 先把整理后的文字写入剪贴板。
- 再通过辅助功能权限模拟 `Command + V` 粘贴。
- 粘贴完成后，可选择恢复原剪贴板内容。

这种方式比直接操作目标 App 的文本控件兼容性更好。

### 3.2 当前没有输入框

如果当前没有可用输入框，例如用户正在：

- 查看设计图
- 浏览代码但没有编辑焦点
- 看网页、PDF、图片或视频
- 切换应用、整理窗口

则浮声不强行输入，而是把文字保存到后台草稿区。

草稿区应支持：

- 最近语音记录列表
- 一键复制
- 一键粘贴到当前焦点
- 删除记录
- 可选的自动标题或时间戳

### 3.3 判断标准

原始设想中提到“鼠标是否在输入框内”，但这个判断不够准确。更合适的判断标准是：

**当前系统焦点是否是可输入文本控件。**

原因：

- 鼠标停在输入框上，不代表输入框已经获得焦点。
- 输入框已经获得焦点时，鼠标可能早已移走。
- macOS 输入行为由系统焦点决定，而不是由鼠标位置决定。

因此，浮声应基于 Accessibility API 检测当前聚焦控件，而不是基于鼠标位置判断。

## 4. 技术实现方案

### 4.1 App 形态

建议做成一个轻量级 macOS 菜单栏 App。

主要界面包括：

- 菜单栏图标
- 录音状态窗口
- 设置页
- 历史草稿页
- 快捷键设置
- API Key 配置

### 4.2 主要模块

#### 全局快捷键模块

用于监听用户配置的语音触发快捷键。

可选实现：

- macOS 原生全局热键 API
- Swift 第三方库，例如 KeyboardShortcuts

建议功能：

- 按一次开始录音，再按一次停止。
- 或者按住说话，松开结束。
- 支持用户在设置中自定义快捷键。

#### 录音模块

使用 macOS 音频能力采集麦克风音频。

可选实现：

- `AVAudioEngine`
- `AVAudioSession` 相关能力
- 音频格式转换为 ASR 服务要求的采样率和编码格式

需要处理：

- 麦克风权限
- 静音检测
- 录音开始/结束状态
- 网络异常时的缓存或重试

#### ASR 识别模块

使用阿里百炼 / DashScope 的语音识别能力。

推荐优先考虑实时语音识别，因为它适合“按快捷键后边说边转写”的输入体验。阿里百炼实时语音识别通过 WebSocket 接收音频流并实时返回转写文本，适合语音助手、会议字幕、实时输入等低延迟场景。

官方参考：

- https://help.aliyun.com/zh/model-studio/real-time-speech-recognition-user-guide
- https://help.aliyun.com/zh/model-studio/fun-asr-realtime-websocket-api
- https://help.aliyun.com/zh/model-studio/non-realtime-speech-recognition-user-guide

后续落地前需要确认：

- 使用哪个 ASR 模型
- 音频格式要求
- WebSocket 鉴权方式
- 是否支持热词
- 是否支持标点、时间戳、多语种或方言
- 价格和并发限制

#### 文本整理模块

ASR 原始结果通常会存在：

- 标点不理想
- 口语重复
- 断句不自然
- 同音词错误
- 专有名词识别不稳定

因此，ASR 结束后可以调用大模型做整理。

整理目标：

- 保留原意
- 修正明显错字
- 补充标点
- 删除口头禅和重复词
- 根据用户选择生成不同风格

可支持的模式：

- 原文模式：尽量保留口语。
- 整理模式：适合发消息、写文档。
- 专业模式：适合写需求、技术说明、会议纪要。
- 简短模式：压缩成更短的表达。

#### 焦点检测模块

使用 macOS Accessibility API 检测当前前台 App 的聚焦元素。

需要判断：

- 当前前台 App 是什么
- 当前 focused element 是否存在
- focused element 是否支持文本输入
- 是否可以接受粘贴

常见可输入控件包括：

- `AXTextField`
- `AXTextArea`
- WebView 中的文本输入区域
- 支持文本编辑的自定义控件

注意：不同 App 的可访问性实现不完全一致，因此不能保证所有输入框都能被准确识别。

#### 文本输入模块

推荐默认使用“剪贴板 + 模拟粘贴”的方式。

流程：

1. 保存当前剪贴板内容。
2. 把识别后的文字写入剪贴板。
3. 模拟 `Command + V`。
4. 可选：延迟后恢复旧剪贴板。

优点：

- 兼容大部分 App。
- 对中文、换行、长文本更稳定。
- 避免逐字模拟键盘导致输入错误。

需要注意：

- 恢复剪贴板可能会影响用户正在复制的内容。
- 可以在设置中提供“是否恢复剪贴板”的选项。

#### 历史草稿模块

当没有输入框时，文字进入草稿区。

草稿数据建议保存：

- 文本内容
- 创建时间
- 是否已复制或粘贴
- 来源 App
- 整理模式
- 原始 ASR 文本

本地保存即可，建议不默认上传历史文本。

#### 设置与密钥管理模块

设置项建议包括：

- API Key
- ASR 模型选择
- 大模型选择
- 快捷键
- 录音触发方式
- 默认整理模式
- 是否自动粘贴
- 是否保留历史记录
- 是否恢复剪贴板

API Key 应保存到 macOS Keychain，不应明文写入配置文件。

## 5. 权限要求

浮声至少需要以下权限：

### 麦克风权限

用于录制用户语音。

### 辅助功能权限

用于：

- 获取当前聚焦元素
- 判断是否存在输入框
- 模拟粘贴快捷键

### 网络权限

用于调用阿里百炼 ASR 和大模型服务。

### 可选权限

如果后续需要读取当前 App 信息、窗口标题或更复杂的上下文，可能还需要额外权限，但 MVP 阶段应尽量减少权限范围。

## 6. MVP 版本建议

第一版不要做太复杂，建议只实现最核心链路。

### MVP 功能

- 菜单栏 App
- 设置 API Key
- 设置全局快捷键
- 按快捷键开始/停止录音
- 调用阿里百炼 ASR
- 调用大模型整理文本
- 如果当前有输入框，自动粘贴
- 如果当前没有输入框，保存到历史草稿
- 历史草稿支持复制

### 暂不做

- 多设备同步
- 团队账号
- 复杂模板
- 复杂上下文读取
- 自动识别 Figma / IDE / 浏览器内容
- 语音命令系统

## 7. 主要风险和限制

### 输入框识别不可能 100% 准确

macOS 不同 App 的可访问性实现差异较大。有些输入区可能不是标准文本控件，因此需要允许用户手动选择：

- 直接粘贴
- 保存到草稿
- 复制到剪贴板

### 模拟粘贴需要辅助功能权限

首次使用时，用户需要到系统设置中授权。这个流程需要在 App 中做清晰引导。

### 剪贴板可能被临时覆盖

如果使用剪贴板粘贴，需要谨慎处理旧剪贴板恢复。建议提供设置项，让用户选择是否恢复。

### 网络 ASR 有延迟和隐私问题

语音和文本会发送到阿里云服务。设置页和首次启动时需要明确告知用户。

### API Key 安全

API Key 必须存放在 Keychain。日志中不能打印 Key，也不能把 Key 写入崩溃报告或普通配置文件。

## 8. 后续增强方向

### 热词

支持用户维护常用词，例如：

- 产品名
- 人名
- 公司名
- 技术词
- 项目代号

用于提升 ASR 准确率。

### 多种输出风格

例如：

- 聊天消息
- 需求文档
- 技术说明
- 会议纪要
- 邮件
- 小红书/公众号文案

### 快捷操作

例如：

- 复制最近一条
- 粘贴最近一条
- 重新整理
- 翻译
- 总结
- 转成待办事项

### 本地识别模式

未来可以考虑接入本地 Whisper 或 Apple Speech，减少网络依赖。

## 9. 推荐结论

这个方案可行，并且比依赖微信输入法语音输入更稳定。

关键设计调整是：

**不要判断鼠标是否在输入框内，而要判断当前系统焦点是否是可输入文本控件。**

最终推荐产品方案是：

**浮声 = 快捷键语音输入器 + 有输入框就自动输入 + 没输入框就后台记草稿 + 用大模型整理文本。**

这套方案适合先做一个轻量 MVP，再逐步扩展为更完整的 macOS 语音输入效率工具。
